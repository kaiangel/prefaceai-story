## 一、创意输入方式升级

### 现状
- 文本输入框（必填）
- 文档上传（PDF/TXT/DOCX）

### 新增输入方式（按优先级）

| 优先级 | 输入方式 | 说明 | 技术依赖 | 工作量 |
|--------|---------|------|---------|--------|
| **P1** | 图片 OCR | 上传截图/图片 → Haiku 提取文字 → 填入故事创意 | Haiku API (vision) | 中 |
| **P1** | 语音输入 | "按住说话" → Whisper STT → 填入文本 | Whisper (已在项目中) | 中 |
| **P2** | 一键故事模板 | 3-5 个可点击 story starters，点击预填到输入框 | 纯前端 | 低 |
| **P3** | 图片情绪参考 | 上传图片作为风格/氛围参考（不提取文字） | LLM vision 分析 | 高 |

**PM 分析**:

- **语音输入强烈推荐** — 目标用户是抖音/快手创作者，语音是他们最自然的表达方式。Whisper 已经在项目中，复用成本极低。移动端体验提升巨大。
- **图片 OCR** — Founder 提出，场景明确（用户截图带文字的聊天/文章作为创意输入）。Haiku vision 能力足够做文字提取。
- **一键模板** — 解决冷启动"不知道写什么"问题。纯前端工作量极小，高性价比。
- **图片情绪参考** — 优先级低，可后做。

## 二、生成流程中的预览与控制

### 核心设计原则: 自动流 + 可选介入

**不做成强制暂停**，而是"自动展示 + 可选暂停"：
- A 型用户（多数）：看到预览的一键流，什么都不点，自动继续
- B 型用户（少数高价值）：每一步都能点击介入

### 检查点 1: 角色预览（Stage 5a 完成后）

**触发时机**: 角色参考图（portrait + fullbody）全部生成完成后

**展示内容**:
- 每个角色的 fullbody 预览卡片（portrait 不展示，但已生成）
- 角色名 + 简短描述

**交互**:
- **默认行为**: 自动倒计时 10 秒后继续下一步（进度条或计时器可见）
- **用户介入**: 点击任意角色卡片的"调整"按钮 → 暂停自动继续
- **调整方式**:
  - 预设修改标签（主推）: "换发色"、"换服装"、"更年轻/更成熟"、"换风格"等
  - 自由输入（辅助）: 底部小输入框，placeholder 引导（如"我想让她穿红色连衣裙"）
- **重新生成**: 点击角色卡片上的重新生成按钮 → portrait + fullbody 重新生成（只展示 fullbody 结果）
- **全局确认**: "继续生成" 按钮（或倒计时结束自动继续）

**PM 分析**:
- 预设标签 > 自由输入。多数用户不知道该写什么 prompt，标签降低摩擦。
- 只展示 fullbody 是对的。portrait 是实现细节。
- 重新生成成本: 每角色 2 次 API 调用（portrait + fullbody）。需让用户感知到"这需要一点时间"。

### 检查点 2: 场景描述确认（Stage 5a.5 之前）

**触发时机**: 角色确认后、场景参考图生成之前

**展示内容**:
- Stage 1 `unique_locations` 的精简文字描述
- 格式: "场景 1: 雨夜公交站 — 昏黄路灯下的积水路面..."

**交互**:
- **默认行为**: 自动倒计时继续（同角色检查点）
- **用户介入**: 点击任意场景描述的"修改"按钮
- **修改方式**: 每个场景描述旁一个小输入框（placeholder: "描述你想要的场景氛围"）
- **修改确认**: 修改后场景描述更新，影响后续场景参考图 prompt
- **全局确认**: "开始绘制" 按钮

**PM 分析**:
- **零成本检查点** — `unique_locations` 数据已有，纯前端展示，不需要额外 API 调用。
- 在花钱生图之前让用户确认方向 — 成本节约型设计。
- 不展示场景参考图预览是正确的（还没生成，用户看文字描述足够）。

### 检查点后: 后台生成 + 用户自由离开（Founder 补充 2026-03-19）

**触发时机**: 角色 + 场景确认后，开始生成 shots（从"正在绘制第 1 张画面"开始）

**Founder 要求**: 用户在 shots 开始生成时可以离开，不需要在那等待。生成在后台进行。

**行为**:
- 进度仍然展示（当前 StageC 的进度条 + 日志）
- **新增**: "后台生成"按钮 — 点击后返回到 Dashboard 或用户想去的任何地方
- Dashboard 中显示生成状态（卡片上显示进度百分比 + "生成中..."标签）
- **通知**: 生成完成后：
  - Dashboard 卡片状态更新
  - 浏览器推送通知（如果用户授权了）
  - 页内 toast 提醒（如果用户在 Dashboard）

**PM 分析**:
- 一个完整故事 18-36 shots，按 ~30s/shot 估算，总生成 9-18 分钟。强制等待是灾难性 UX。
- 后台生成是 AI 生成类产品的标配（Midjourney、Runway 等都这样做）。
- **生成失败处理**: 如果某个 shot 生成失败但用户已离开，Dashboard 卡片需显示"生成异常"状态 + 重试入口。
- **部分结果流式展示**: 未来优化方向 — 已完成的 shots 可以先展示，用户可以边看边等。当前 MVP 先做完整完成后才展示。

## 三、Dashboard / 用户控制台（Founder 补充 2026-03-19）

### Founder 要求

用户需要一个 Dashboard/控制台，包含：
- **"我的作品"** — 故事列表 + 生成中状态提醒
- **会员订阅入口** — 含会员状态/等级/积分 Credits 消耗情况/个人资料的设置
- **故事详情功能** — 续作、做同款、导出素材/合成视频、页面内播放、分享、收藏、排序、删除
- 以及这些功能涉及到的交互中包含的完整延展元素、功能或页面

### 现状
已有基础 Dashboard:
- `/dashboard` — 故事列表 + 统计卡片（总数/已完成/总画面数）+ 问候语
- `/dashboard/[storyId]` — Shot 轮播 + 旁白 + 角色/风格/日期信息
- 组件: StoryCard, StoryGrid, EmptyState, UserMenu
- 已有: 续写按钮(mock)、下载按钮(mock)、删除(mock)

### 需要补充的功能

**Dashboard 主页 (`/dashboard`)**:
| 功能 | 现状 | 需补充 |
|------|------|--------|
| 故事列表 | ✅ StoryGrid | - |
| 统计卡片 | ✅ 总数/完成/画面数 | 补: Credits 余额卡片 |
| 生成中提醒 | ❌ | 顶部 banner: "《xxx》正在生成中... 67%"，点击跳转查看进度 |
| 排序 | ❌ | 最新/最早/名称 |
| 会员入口 | ❌ | UserMenu 下拉中 or Header 独立按钮 |
| 新建故事 | ✅ | - |

**故事详情页 (`/dashboard/[storyId]`)**:
| 功能 | 现状 | 需补充 | 说明 |
|------|------|--------|------|
| Shot 轮播 | ✅ | - | |
| 旁白显示 | ✅ | - | |
| 导出素材 | ✅ mock "下载" | 细化为: 导出图片+音频打包下载 | 用户二次创作用 |
| 合成视频 | ❌ | 触发视频合成 + 下载 | 用于直接发布到抖音等 |
| 续作 | ✅ mock "续写" | "按现有故事情节续作" — 同角色同世界观继续剧情 | 技术复杂（需传递角色+上下文） |
| 做同款 | ❌ | "做同款故事" — 预填 style + 角色设定，故事创意全新 | 纯前端路由跳转 |
| 页面内播放 | ❌ | Shot 序列自动播放 + 背景音乐（如有） | 模拟视频效果 |
| 分享 | ❌ | 生成分享链接 / 二维码 / 一键转发社交平台 | |
| 收藏 | ❌ | 收藏/取消收藏标记 | |
| 排序 | ❌ | 简单排序（按时间/名称） | |
| 删除 | ✅ | 带确认弹窗 | |

**设置页 (`/settings` 或 Dashboard 子路由)**:
| 功能 | 现状 | 需补充 |
|------|------|--------|
| 个人资料 | ❌ | 头像/昵称/邮箱/手机号 |
| 会员状态 | ❌ | 等级/到期日 |
| Credits | ❌ | 余额 + 消耗历史（每个故事的成本明细） |
| 订阅管理 | ❌ | 升级/续期/取消订阅 |

**PM 分析**:
- **生成中提醒是关键** — 用户离开生成页后，Dashboard 必须清晰显示生成状态。否则用户会焦虑"我的故事到哪了"。
- **续作 vs 做同款** — 这是两个截然不同的功能:
  - **续作**: 同角色 + 同世界观 + 从上一个故事结尾继续。需要传递完整的角色数据 + 故事上下文给 Stage 1。技术上需要后端支持"继承角色数据"的 API。**建议 MVP 只做"做同款"**，续作技术复杂度高。
  - **做同款**: 预填 style preset + 角色数量/类型到 Create 页，但故事创意全新。纯前端路由跳转 + 参数传递，工作量低。
- **Credits 系统** — 这是商业化基础。前端可以先做 UI mock，但**定价策略需要 Founder 定**（多少 credits/故事、不同篇幅/风格成本差异、免费用户试用次数）。
- **页面内播放** — 把 shot 图片按顺序自动切换 + 配旁白音频（如有），模拟视频效果。纯前端可实现（shot 图序列 + 定时器）。
- **完整延展元素** — 分享需要: 分享链接生成页 + 二维码弹窗 + 社交平台选择器。合成视频需要: 进度弹窗 + 下载触发。这些都是独立的 UI 组件/弹窗。

## 四、画面比例

### 研究结果

| 平台 | 内容形式 | 推荐比例 | 具体尺寸 |
|------|---------|---------|---------|
| **抖音** | 短视频 | 9:16 | 1080×1920 |
| **抖音** | — | **2:3 已确认可用** | 1080×1620 |
| **小红书** | 视频笔记 | 9:16 / 3:4 | 1080×1920 / 1080×1440 |
| **小红书** | **图文笔记** | **3:4 (最佳)** | **1080×1440** |
| **小红书** | 图文笔记 | 1:1 | 1080×1080 |

**关键发现**: 小红书图文笔记的黄金比例是 **3:4** (1080×1440)，竖版封面显示面积比横版多 ~40%。而我们当前的 2:3 (1080×1620) 比 3:4 更窄更高。

### Founder 补充 (2026-03-19)

- 抖音 2:3 已确认合适
- 小红书有两种形态：视频 + 图文，需分别确认

### PM 研究结论 + 建议

**小红书图文黄金比例是 3:4** (1080×1440)，竖版封面显示面积比横版多 ~40%。视频笔记 9:16 或 3:4 都可。

**Create 页面的比例选择器**:

| 选项 | 比例 | 像素 | 适用场景 | 默认？ |
|------|------|------|---------|--------|
| 抖音竖屏 | 2:3 | 1080×1620 | 抖音短视频、小红书视频 | ✅ 默认 |
| 小红书图文 | 3:4 | 1080×1440 | 小红书图文笔记（占屏面积最大） | — |
| 方形 | 1:1 | 1080×1080 | 通用、微信朋友圈 | — |

- 2:3 保持默认（已验证，全链路支持）
- 3:4 作为新增选项（后端 ImageGenerator 的 `aspect_ratio` 参数已支持自定义字符串，可直接传 "3:4"）
- 1:1 可后加
- **注意**: 当前全链路（参考图、场景图、shot 图、TextOverlay）硬编码了 `"2:3"`。支持 3:4 需要全链路参数传递，在接真实 API 时处理。Mock 阶段先做 UI 选择器。

## 五、实施策略

**全部 mock 数据先行**（Founder 明确要求），迭代 UX 后再接真实 API。

### 建议分批（Founder 已同意优先级）

**Batch 1 (P0 — 角色/场景预览流)**:
- StageC 拆分: 生成进度 → 角色预览检查点 → 场景描述检查点 → 继续生成
- 自动继续倒计时 + 可选暂停
- 角色调整 UI（预设标签 + 自由输入 + 重新生成按钮）
- 场景描述展示 + 修改输入
- Mock: 角色 fullbody 图片 + 场景文字描述

**Batch 2 (P0 — Dashboard 补全 + 后台生成)**:
- "后台生成"按钮 → 用户自由离开
- Dashboard 生成中状态展示（卡片进度条 + banner 提醒）
- 故事详情页补全: 做同款、页面内播放、分享、收藏、排序
- 导出素材 + 合成视频入口
- 设置页: 个人资料 + 会员状态 + Credits

**Batch 3 (P1 — 输入方式)**:
- 图片 OCR 上传（Haiku vision 提取文字）
- 语音输入（Whisper STT，按住说话）
- 一键故事模板（3-5 个 story starters）

**Batch 4 (P1 — 商业化 UI + 比例)**:
- 会员/Credits 完整 UI（定价策略需 Founder 定）
- 订阅管理页
- 比例选择器（3:4 小红书图文选项）

**Batch 5 (P2 — API 对接)**:
- 全部 mock → 真实 API 切换
- 每个环节逐步替换
