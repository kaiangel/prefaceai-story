# Agent Team PM 核心方法论（通用版）

> 给任何用 Agent Team / 多人协作开发的项目 PM。两大块：① 地毯式深度审查铁律 ② 跨团队协作 + 架构纪律。
> 已脱敏具体项目技术，保留可迁移的方法论 + 通用实证教训。

---

## 一、地毯式深度审查铁律

> **核心信条：验证"真的做到了"，不是"看起来做了"。**
> 同事 / Agent 完成工作后，PM 必须独立深度审查 —— **不能只信自报、不能只看代码、不能只 grep 存在**。

### 铁律 1：grep "存在与否" 远远不够 —— 追完整调用链路
- 函数定义 ≠ 函数被调用（grep 到定义，必须再 grep 调用点确认链路接通）
- 字段加进 schema ≠ 字段被消费（追到消费端是否真读取）
- 参数加进签名 ≠ 参数被传值（查实际调用处传了没）
- 完整链路要追：**定义 → 调用点 → 参数传递 → 数据流向 → 消费点**，每一环都查
- **实证**：某 agent 报"5 项全完成 + 测试通过"，第一轮只 grep 验证函数定义存在就差点放过；深挖发现该函数 **0 次调用 = 死代码**，功能根本没修。

### 铁律 2：不信自报 —— PM 独立实测验证
- Agent 报"完成 / PASS / 通过"，PM 必须**自己跑一遍验证**，不是看汇报
- **代码审查通过 ≠ 运行时生效**。实证：一处修复的正则在字符串构造层（模板字符串）被转义破坏，源码看着完全正确，但运行时 0 生效；只有自己跑 e2e、看真实运行日志才发现。
- **自动化验证器判通过 ≠ 人工无瑕疵**。实证：视觉验证器判某产出"通过"，但人眼复核仍有明显缺陷 —— 验证器有盲区，关键产出仍需人工复核。

### 铁律 3：实测纠正纸面判断
- **不拿"看到的字符串"反推根因**。实证：看到请求超时就推断"查询拉了全表"，实测（测网络往返延迟）发现根因是网络层延迟，跟"全表"完全无关 —— 拿现象反推会错。
- 用实测工具检验：跑测试 / 测接口耗时与状态码 / 看产出物的实际属性（如文件真实尺寸）/ 测网络延迟 / 看真实日志 / 看实际产物
- 纸面判断必须被实测检验；错了就**透明纠正**（不藏自己的误判）

### 铁律 4：对照任务清单逐条核对
- "已修复"必须对照**原始要求逐条**核对（要求"加 X 缓冲"，结果写了 `>=` 没缓冲 = 没做到）
- 不放过任何一条验收标准

### 铁律 5：工作量越大越要地毯式
- 改动越多 / 跨文件越多 / 跨模块越多 → 越容易漏
- 大批量修复 / 架构改动 / 跨前后端数据流 = 最高风险，最要逐条地毯式

### 铁律 6：跨边界数据流最容易断
- 前后端 / 模块间的字段名、接口契约，**两端都要查对齐**（一端改了另一端没跟 = 断裂）
- **实证**：后端把一个接口从同步改成异步（返回任务 ID 让前端轮询），但前端还按同步方式调用（期待直接拿结果）→ 功能直接断；只有地毯式查两端契约才发现这个"单方改契约"的 gap。

### 触发时机
任何 Agent / 同事完成工作后 —— 尤其大批量修复、架构改动、跨多文件协作、跨前后端数据流。

## 二、跨团队协作 + 架构纪律

> 来自资深 CTO 的实践原则，适用于多人 / 多团队 / 前后端协作。这些直接命中"多人协作高频 bug"的根因。

### 原则 1：前后端纠验机制（最高价值）
- **单一权威状态源**：一端（通常后端）计算权威状态，另一端（前端）**只派生、不本地猜测、不本地缓存**
- **契约文档化**：接口契约写成正式文档；**改契约必须先改文档，再改代码**
- **工具强制**：pre-commit hook 强制每个 commit 标注"是否影响对端"（如 `frontend-impact: yes/no`）
- **根源**：实测某轮端到端测试，47% 的 bug 都是"前后端契约断裂"（一端改了另一端不知道）—— 这套机制直接命中根因，后续同类 bug 修复速度快数倍。

### 原则 2：共享数据层纪律
- 开发 / 测试 / 生产连**同一共享数据源**，不自建本地副本
- 避免"本地能跑、生产挂"的环境不一致掩盖问题

### 原则 3：跨领域改动事先沟通
- 碰别人领域（后端 / 数据库 / 架构）前**先打招呼**，不闷头改
- 各人守自己领域边界；不确定归属先问，不自行越界

### 原则 4：部署 / 同步纪律
- 改动 **push 到共享处后才通知**协作方（否则对方看不到）
- 部署方式团队约定一致（如统一某种方式，不混用）
- **线上先保持已验证的稳定版**，新代码先本地 / 预发验证通过，再推线上（不把没验证的代码推生产）

### 原则 5：变更可追溯
- commit message 覆盖**完整范围**；一个 commit 含多批改动时全部标注，不漏

## 三、PM 如何落地（审查 SOP）

每次同事 / Agent 通知完成 →
1. **重读协作记录**（群聊 / 进度文档，对方可能追加了大量内容）
2. **逐个读改动的每个文件**，独立深入检查
3. **追调用链路**（铁律 1）：定义→调用→传参→数据流→消费点
4. **对照验收标准逐条**（铁律 4）
5. **独立实测验证**（铁律 2/3）：跑测试 / 实测接口 / 看真实产出，不信自报
6. **查跨边界契约**（铁律 6）：前后端 / 模块两端对齐
7. **查越权**：是否改了不属于自己的文件
8. **查协作维度**（原则 1-5）：契约一致性 / 共享数据 / 是否事先沟通 / frontend-impact 标注

**审查结论无论通过与否，先记录（文档）再进下一步。** 发现问题直接要求修复（修复循环建议上限 2 轮，超限上报决策者）；跨架构 / 边界的判断 → 上报决策者，不自行拍板。

---

## 一句话总结

> **审查：验证"真做到了"而非"看起来做了" —— 追调用链路、不信自报、实测纠正纸面、跨边界两端对齐。**
> **协作：单一权威源 + 契约文档化 + 工具强制标注影响面 —— 直接掐死"一端改了另一端不知道"这个多人协作最高频的 bug 源。**
