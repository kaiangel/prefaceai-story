# 测试驱动迭代，直到全绿

> 测试不是验收，是设计。先写测试，是在定义"正确"长什么样。
> 全绿不是终点，是底线。

---

## 三种模式

根据 `$ARGUMENTS` 的形式自动判断模式：

| 输入 | 模式 | 含义 |
|------|------|------|
| `/xhtdd 实现确认大纲的数据校验` | **完整模式** | 先写测试，再写实现，迭代到全绿 |
| `/xhtdd fix app/api/projects.py` | **修复模式** | 已有代码和测试，跑测试 → 修实现 → 迭代到全绿 |
| `/xhtdd verify tests/test_confirm_outline.py` | **验证模式** | 只跑指定测试，失败就修，不写新测试 |

---

## 执行流程

### 第零步：侦察

1. 识别项目的测试框架：
   - 检查 `pytest.ini` / `pyproject.toml` / `setup.cfg` → **pytest**
   - 检查 `package.json` 中的 test script → **jest / vitest**
   - 检查目录结构 `tests/` / `__tests__/` / `*.test.*`
2. 确定测试运行命令（如 `pytest -v`、`npm test`、`npx jest` 等）
3. 理解已有的测试组织结构和命名惯例

### 第一步：写测试（完整模式独有）

在写任何实现代码之前，先在合适的测试文件中编写全面的测试：

**必须覆盖**：
- **正常路径** — 标准输入、预期输出
- **边缘情况** — 空字符串、None、空列表、超长输入
- **Unicode/编码** — 中文字符、繁简混合、特殊符号、emoji
- **类型校验** — Pydantic 模型验证、字段缺失、类型错误、额外字段
- **边界值** — 0、负数、最大值、刚好超过阈值
- **集成点** — 与现有 pipeline 阶段的交互是否正确

**命名惯例**：跟随项目已有的测试命名风格（`test_*.py` / `*_test.py`）

### 第二步：写实现（完整模式独有）

基于测试定义的"正确"，编写实现代码。

### 第三步：跑测试

```bash
# 自动选择正确的命令
pytest -v {相关测试文件} 2>&1
# 或
npm test -- --verbose 2>&1
```

- 如果是完整模式或修复模式：跑相关测试文件
- 如果是验证模式：跑 `$ARGUMENTS` 指定的测试

### 第四步：读失败 → 修实现 → 重跑

如果有测试失败：

1. **逐条读失败输出**，理解每个失败的根本原因
2. **修实现代码**，不修测试 — 除非测试本身有明确的逻辑错误（如断言值写错）
3. **重新跑测试**
4. 重复，直到全绿

**安全上限：最多 5 轮迭代**。第 5 轮后仍有失败：
- 停止修改
- 输出当前失败的测试列表和根因分析
- 建议 Founder 人工介入

### 第五步：全绿后清扫

测试全部通过后，扫描本次修改的所有文件：

```bash
grep -rn 'TODO\|HACK\|FIXME\|XXX\|WORKAROUND' {本次修改的文件列表}
```

如果发现临时性代码：
- 能立即解决的 → 修掉，重跑测试确认仍然全绿
- 需要后续处理的 → 保留标记，在输出摘要中列出

### 第六步：输出迭代摘要

```
## 迭代摘要

- **模式**：完整 / 修复 / 验证
- **测试框架**：pytest / jest / ...
- **迭代轮数**：N 轮
- **最终结果**：全绿 ✅ / 仍有失败 ❌（剩余 N 个）

### 每轮记录
| 轮次 | 失败数 | 修复内容 |
|------|--------|----------|
| 1    | N      | {简要描述} |
| ...  | ...    | ...      |

### 修改的文件
- {文件路径} — {改了什么}

### 残留 TODO/HACK
- {如有}

### 最终测试输出
{粘贴最后一轮的测试输出}
```

## 铁律

1. **先测试后实现** — 完整模式下，实现代码在测试之后写，不能反过来
2. **修实现不修测试** — 测试是规格说明书，失败说明实现有问题。唯一的例外是测试本身有明确的逻辑 bug
3. **不跳过失败** — 不用 `@pytest.mark.skip`、`xit`、`.skip` 绕过失败
4. **不静默异常** — 不用空的 `except: pass` 让测试假绿
5. **改了高风险文件必须全跑** — 如果本次修改涉及 `image_generator.py`、`storyboard_prompts.py` 等高风险文件，最后一轮必须跑完整测试套件而非只跑相关测试
